Overview

What Argus does

Argus is an AI observability platform with security built in from the ground up. It records every trace your LLM application produces — every prompt, tool call, retrieved document, and model response — the same way a conventional observability tool (like Langfuse or Datadog) would. On top of that, every piece of content that flows through your application is analyzed by a layered detection pipeline that looks for prompt injection, jailbreaks, data exfiltration, and other attacks that are unique to LLM applications.

The core idea: most security tools only look at a single message in isolation. Argus looks at the whole trace — what your agent read, what it did next, and whether its behavior changed after reading untrusted content. That's what makes it possible to catch indirect prompt injection: an attack where instructions are hidden inside a document, web page, or tool result rather than typed by the end user.

What you can use it for

  • Debugging & performance — see exactly what your agent did on any request: every span, every tool call, every token, every dollar spent, and how long each step took.
  • Threat detection — get alerted when a user or a piece of retrieved content tries to hijack your agent, exfiltrate data, or bypass your system prompt.
  • Incident response — when something goes wrong, open the trace, see the taint-tinted waterfall, and find the exact span and evidence that explains what happened.
  • Analyst triage — work through the Review Queue, confirm or dismiss flagged events, and keep a clean record of what's a real attack versus noise.
  • Cost & usage tracking — see spend and token usage broken down by model, provider, and environment.
  • Compliance evidence — because full, untruncated content is stored for every observation (not just previews), you have a complete audit trail to show what your AI system actually did.
Who this is for: AI/platform engineers who want full observability into their LLM app, and security engineers who need AI-specific threat detection — without stitching two separate tools together.

Core concepts

A few terms appear throughout the app. Understanding them makes every page easier to read.

Traces, spans & observations

A trace is one end-to-end run of your application — for example, one user turn in a chat, or one agent task. A trace is made up of observations (also called spans): individual steps like a model call, a document retrieval, or a tool invocation. Argus reconstructs the full sequence and timing of these steps for every trace.

Span typeIconMeaning
generationGA call to a language model (prompt in, completion out).
retrievalRA document or chunk fetched from a knowledge base, search index, or the web.
toolTA function/tool call the agent made (e.g. send_email, get_customer_profile).
spanSA generic step, such as the overall request handler.
eventEA point-in-time marker with no duration.

Taint — trust classification

Every span is automatically classified by how trustworthy its content is. This is the foundation of indirect-injection detection: an instruction is only dangerous if it comes from somewhere the model shouldn't blindly obey.

Taint classAssigned to
systemYour system prompt / developer instructions — fully trusted.
userText typed by the end user.
untrusted_externalRetrieval and tool spans, by default. Anything fetched from outside your app — documents, search results, emails, web pages, other tools — is treated as untrusted unless explicitly marked otherwise.
modelText generated by the model itself (a completion).

In the trace waterfall (see below), spans tainted untrusted_external are washed orange, and any span downstream of one is marked taint-influenced — meaning it may have been affected by content your app didn't fully control.

Detection layers (L1–L4)

Every trace is scanned by up to four layers, each cheaper and more targeted than the last:

LayerWhat it checksRuns on
L1 — HeuristicsRegex & structural rules: instruction-override phrases, obfuscated/invisible characters, exfiltration patterns.100% of content, in microseconds.
L2 — ClassifiersMachine-learning models that score text for injection/jailbreak likelihood. Optional — only active if enabled on your deployment.Untrusted content and anything L1 flags.
L3 — LLM judgeA language model asked to judge whether text is trying to instruct an AI system. Optional — requires a judge endpoint to be configured.Only content already escalated by L1/L2.
L4 — Trace analysisLooks across the whole trace: did untrusted content's instructions get echoed into a later tool call? Did data flow out to an unexpected destination? Did a canary token leak?Runs once a trace is complete.
If Detection layer activity on the Threat Center always shows 0 for L2 and L3, that's expected — those two layers are optional and simply aren't enabled on every deployment. L1 and L4 always run.

Severity, outcome & category

criticalhighmediumlowinfo

Severity reflects how serious a finding is, from info up to critical. Outcome tells you what happened as a result:

OutcomeMeaning
attemptedThe content looked like an attack, but there's no evidence it changed the agent's behavior.
succeededThe attack appears to have worked — e.g. the agent's later actions echoed the injected instruction, or data actually flowed out.
blockedThe attempt was stopped before it could take effect.
unknownNot enough information to say either way.

Category classifies the type of finding: direct_injection, indirect_injection, jailbreak, exfiltration, excessive_agency, rag_poisoning, prompt_leak, pii_egress, canary_triggered, and obfuscation.

Analyst verdict

Every event starts as unreviewed. When you click Confirm malicious or False positive anywhere in the app, that verdict is permanently recorded on the event. Confirmed and dismissed events disappear from the Review Queue but remain visible everywhere else, tagged with a small label:

ConfirmedFalse Positive

Quick start

The dashboard is read-only against your data — the real setup work is making sure your application is sending traces to Argus. Here's the typical path from zero to a working dashboard.

1
Get an API key

Self-service — no login required. Use to create a project and get a public/secret key pair in under a minute. You'll authenticate ingestion requests with HTTP Basic auth using this pair.

2
Point your app at the ingestion endpoint

Send traces to POST /api/public/ingestion (a native/Langfuse-compatible batch endpoint) or POST /v1/traces (OpenTelemetry GenAI-convention spans). Most existing OTel or Langfuse instrumentation works with little to no change — the onboarding flow shows a ready-to-run snippet for each.

3
Let taint classification happen automatically

You don't need to tag anything manually — retrieval and tool spans are treated as untrusted by default. If a specific tool is actually safe (e.g. an internal calculator), your admin can add a tool override so it isn't treated as an attack surface.

4
Watch the Overview page

Once traces start arriving, fills in with real counts — traces, spans, tokens, cost, and any security events found.

5
Triage anything flagged

Open to see the live attack feed, or to work through unreviewed events one at a time and record a verdict.

6
Investigate a specific incident

Click any event or trace ID to open — the taint-tinted waterfall shows exactly which span introduced untrusted content and what happened afterward.

Connect a new app

A standalone, three-step self-service page — + Connect a new app in the left rail — for onboarding yourself or a client with no login and no manual database work.

StepWhat happens
1. Name your projectEnter a company/organization name and a project name (e.g. one project per environment: "production", "staging"). Creates the organization, project, a permissive default detection config, and an API key pair.
2. Your API keyThe public and secret key are shown once — Argus stores only a hash of the secret. Copy buttons for each; a warning makes clear this is the only time the secret is shown in full.
3. Test your connectionReady-to-run, pre-filled code snippets for cURL, Node.js, Python, and OpenTelemetry — each with numbered plain-language steps for where the code goes and how to run it. Below it, a live status indicator polls every few seconds and flips from "Waiting…" to "Connected" the moment the test message arrives, no page refresh needed.

On success you get a personalized dashboard link (?project=<id>) — bookmark and share it. Opening that link scopes every page (Overview, Threat Center, Traces, everything) to just that project's data, so one client never sees another's traces or security events.

Your app doesn't need to be on the computer you're testing from. Step 3's test message is just a network call to Argus — run it from any machine with internet access, even if your real application is deployed on Azure, AWS, a VM, Railway, Vercel, or anywhere else. It only proves your API key and network path work; it doesn't touch your real app at all. To connect the real app, add the same kind of tracer code into its actual source code and redeploy it however you normally do (git push, your platform's deploy command, a service restart) — no special access to Argus is needed on that server.

Note: this is intentionally lightweight — the secret key is the only credential, there's no rate limiting or email verification on project creation yet. Treat the onboarding URL like you would any other unauthenticated signup form.

Threat Center

The security nerve center. Everything here is about what's attacking your application right now — the live feed of every finding, plus a look at how the detection pipeline itself is performing.

KPI row

Same shape as Overview's security tiles (Security events, Critical, Injections, Exfiltration), plus two Threat-Center–specific tiles: Canary triggers (a registered secret token was found somewhere it shouldn't be — the strongest possible signal of a real leak) and Taint coverage.

Attack feed

A live, most-recent-first table of every security event. Click any row to expand it and see the full evidence and take action:

Column / elementWhat it shows
Colored stripeMatches the row's severity color — scan the left edge of the table to spot critical rows fast.
SevSeverity pill.
CategoryThe finding type; a verdict tag appears here once you've reviewed it.
Outcomeattempted / succeeded / blocked / unknown pill.

Detection layer activity

Shows how many of the events in your selected range each layer contributed to. A single event can involve more than one layer, so these numbers won't sum to the total event count.

Escalation funnel

The pipeline's volume at each stage: total spans scanned → how many L1 flagged → how many escalated to L2 → how many were judged by L3 → how many actual events were raised. A healthy pipeline narrows sharply — most content is clean and gets filtered out early.

Most-targeted surfaces

Which specific tools or retrieval sources are producing the most security events — useful for spotting a single poisoned knowledge-base source or a consistently abused tool.

By severity / By category

Simple distribution bars so you can see the shape of your threat landscape at a glance.

Events over time

A stacked hourly chart split into three lanes: indirect injection (the most dangerous class), direct injection/jailbreak attempts, and output-side findings like PII egress. Use it to spot spikes or a sustained campaign.

Incidents

Threat Center shows individual events; Incidents groups them into something you can act on.

Recurring / poisoned sources

Groups events by the content hash of the source material, so if the exact same document or web page triggers findings across multiple different traces, it surfaces here as one row instead of many scattered alerts. This is how you catch a single poisoned document in your knowledge base before it affects more users. These rows are informational (not clickable) since a hash spans many traces — use the categories and evidence shown to identify the source and remove it.

Trace incidents

Every trace that contains at least one high or critical event, most severe first. Each card shows the categories involved and a snippet of evidence. Click a card to open that trace directly.

Review Queue

Your working queue: every event that still needs a human verdict, highest severity first. The goal is to empty this queue regularly, so nothing sits unreviewed.

ColumnMeaning
Sev / CategorySame as the attack feed.
EvidenceA preview of the matched content, so you often don't need to open the trace to make a call.
TraceClick to open the full trace for more context.
Confirm / DismissSame verdict actions as Threat Center, in compact form. As soon as you click one, that row disappears from the queue (it's been reviewed) and the badge count on the left rail updates.

When the queue is empty, you'll see a "Queue clear" message — that's the goal state.

Red Team

Reserved for scheduled, automated attack simulations — running probe suites (like garak or promptfoo) against your application on a schedule and tracking how your attack-success-rate changes over time as you harden your app. This page is currently empty in every deployment; it's a roadmap item. When runs are configured, they'll appear here as scored traces.

Trace detail

The deepest view in the app — reconstructs one trace's full execution as a waterfall, with a side panel to inspect any individual span. You land here by clicking a trace ID anywhere in the app; the ← Back button returns you to wherever you came from.

The waterfall

One row per span, ordered by start time, with a horizontal bar positioned and sized to scale against the trace's total duration. Row and bar coloring tell the security story at a glance:

VisualMeaning
Orange wash + left borderThis span is the untrusted source — e.g. a document retrieval or tool result classified as untrusted_external.
Faint orange washTaint-influenced — this span happened after untrusted content entered the trace, so it may have been affected by it.
Red wash + left borderThis exact span has a high or critical security finding attached — the strongest visual flag, and it overrides the other two.
Bar color: tealNormal — no security finding on this span.
Bar color: orangeThis span has a high-severity finding.
Bar color: redThis span has a critical finding.
Chips next to the span nameUp to two L4 signal names (e.g. exfil_flow, instruction_echo) found on that span.

The taint legend beneath the waterfall recaps these three washes. Click any row to load it into the drawer on the right — the most severe span is auto-selected the moment you open a trace, so the most important thing to look at is already in front of you.

Span drawer (right panel)

Four tabs, once you've selected a span:

TabShows
SecurityEvery security event on this specific span — severity, category, outcome, evidence, score, and which rules/signals fired — with Confirm / False positive buttons for each one. The tab label shows a count when events exist.
InputThe full input payload — the prompt sent to a model, or the arguments passed to a tool — rendered as plain, non-executing text (safe to view even if it contains an attack payload).
OutputThe full output payload — a model's completion, a tool's result, or a retrieved document's content.

Why this is safe to browse: everything in the Input/Output tabs is rendered strictly as text — never executed or interpreted as HTML — so even a document containing a prompt-injection or script payload can never affect your browser. Argus is designed to store and display attack content without becoming vulnerable to it.

Prompts

Reserved for versioned prompt management — storing your system/user prompt templates with labels (e.g. production, staging) and diffing changes over time. No prompts have been registered on this deployment yet; this is a roadmap feature (Phase 3). Once your app registers prompts through the prompts API, they'll appear here.

Evals

Aggregates any scores recorded against your traces — whether from an LLM-as-judge evaluation or human annotation. Each row is one score name (e.g. helpfulness, sec.injection_risk) with its count, average, min, and max across all recorded values. If this is empty, no scores have been submitted yet — they populate as your eval pipeline or annotators write to the scores API.

Connecting your app & admin configuration

These are configured by whoever operates your Argus deployment, not from within the dashboard itself — included here so you understand where the data and detections actually come from.

Sending data in

  • Native batch endpoint — POST /api/public/ingestion, authenticated with HTTP Basic auth using your project's public/secret API key pair. Compatible with Langfuse-style batch payloads.
  • OpenTelemetry endpoint — POST /v1/traces, accepting standard OTel spans using GenAI semantic-convention attributes. If you already instrument your app with OpenTelemetry, this usually requires little to no change.

Taint overrides

By default every retrieval/tool span is treated as untrusted. If a specific tool is actually internal and safe (e.g. a calculator or your own database), an admin can add a tool override in the project's detection configuration so it's no longer treated as an attack surface — reducing false positives.

Alerting

When an event crosses a configured severity threshold, a webhook notification can fire to Slack, PagerDuty, or any endpoint. The destination URL and minimum severity are set by an admin at deploy time; there's no in-app control for this yet.

Detection tuning

The specific rules (L1), whether classifiers/judge (L2/L3) are enabled, and severity thresholds are configured per project by whoever operates the deployment. If a rule is producing too many false positives, that's tuned at the deployment level, not per-analyst — use False positive verdicts in the meantime to keep your own view accurate; those verdicts are recorded for future tuning.

Data retention

Argus stores full, untruncated input/output content for every observation (not just a preview) specifically so that historical traffic can be re-analyzed as detection improves, and so investigations have complete evidence. How long that data is kept is a retention setting controlled by whoever operates your deployment.

FAQ

I don't see any data anywhere.

Either no traces have been ingested yet, or your selected time Range is excluding them. Try switching Range to All time first, and confirm your application is actually sending data to the ingestion endpoint.

Detection layer activity always shows 0 for L2 and L3.

Expected on most deployments — those two layers are optional. L1 heuristics and L4 trace analysis always run and are usually sufficient to catch the events you'll see in the feed.

What exactly happens when I click Confirm / False positive?

Your verdict is permanently recorded on that event. It's removed from the Review Queue (since it no longer needs review) but still visible everywhere else — attack feed, incidents, trace detail — with a small verdict tag, so nothing is hidden or deleted.

My appearance changes disappeared after reloading.

Expected today — see the note in Settings above. Preferences are per-tab only until persistence ships.

Can I search for a specific trace by ID?

Not yet from a search box — open it from any list it appears in (Traces, the attack feed, Incidents, or Review Queue). If you already know a trace ID, an admin can query it directly via the trace API.

Is it safe to view content that contains an attack payload?

Yes. Every piece of stored content — including anything an attacker tried to inject — is rendered strictly as plain text, never executed or interpreted as HTML. A stored prompt injection can never become a stored script in your browser.

Why do Project and Env look like buttons but don't do anything?

The Env selector is still a placeholder. Project now reflects real scoping: it shows "All projects" by default, or a project ID when you're on a personalized onboarding link — see . Neither is a clickable dropdown yet for switching projects manually mid-session.

I onboarded a new project but its dashboard looks empty.

Make sure you're on your personalized link (?project=<id>, shown after your first trace lands) and that your app is actually sending traces to the ingest URL and key pair from step 3 of onboarding. The unscoped dashboard (no ?project=) shows every project combined, so if you see data there but not on your scoped link, double-check the project ID in the URL matches the one you were given.

My app is already deployed on Azure/AWS/a VM/Railway/Vercel — I don't have it running on my computer. How do I connect it?

Two separate things are going on here, and it helps to keep them apart. Testing the connection (onboarding step 3) is just a network call to Argus — it works from any machine with internet access, so your own laptop is perfectly fine even though your real app lives elsewhere. It doesn't touch your real app at all. Connecting your real app is a different, later step: add the same kind of tracer code into that app's actual source code, then redeploy it through whatever process you already use for that platform — a git push, your cloud provider's deploy command, or a service/container restart. You don't need terminal or SSH access to the production server specifically for Argus; you're just adding a few lines of code to a file that gets deployed as part of your normal release process.

I saved the Node.js/Python test file and got a syntax error mentioning \rtf1\ansi or similar.

Your text editor saved the file as Rich Text instead of plain code — on a Mac, this almost always means TextEdit, which defaults to Rich Text even when the file is named .js or .py. Fix: open the file in TextEdit, go to Format → Make Plain Text (Shift+Cmd+T), save again, and re-run it. To avoid this entirely, create the file from a terminal instead: run nano argus-test.js (or argus_test.py), paste the code, then press Ctrl+O, Enter, Ctrl+X — nano always saves plain text.
